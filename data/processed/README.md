# Processed invoice matching datasets

Generated by:

```text
notebooks/create_noisy_invoices.ipynb
```

## Files

- `invoices_noisy_matching.csv`: flat tabular dataset for invoice matching.
- `invoices_noisy_matching.jsonl`: one JSON object per line with both the original nested invoice and the noisy nested invoice.
- `invoices_ground_truth_corpus.jsonl`: 100% of the ground-truth invoices, intended as the embedding/search corpus.
- `invoices_noisy_evaluation.csv`: 30% noisy query set for evaluation, split into 15% validation and 15% test.
- `invoices_noisy_evaluation.jsonl`: nested version of the same noisy evaluation query set.
- `embedding_corpus.jsonl`: embedding-ready corpus generated from `ground_truth` invoices.
- `embedding_queries.jsonl`: embedding-ready query set generated from `noisy_invoice` records, with `evaluation_split` preserved in metadata.
- `corpus_embeddings.jsonl`: corpus records with generated embedding vectors.
- `query_embeddings.jsonl`: query records with generated embedding vectors.
- `semantic_search_results.jsonl`: local semantic search results, one query per row with top-k corpus candidates.
- `semantic_search_metrics.json`: summary metrics for the local semantic search baseline.

The original raw file in `data/raw/invoices_dataset.json` is not modified.

## Column conventions

- `gt_*`: ground-truth values parsed from the original invoice JSON.
- `noisy_*`: synthetic noisy values that simulate OCR, manual entry, and system-format differences.
- `noise_level`: `low`, `medium`, or `high`.
- `error_types`: pipe-separated list of synthetic errors applied to the record.
- `changed_fields`: pipe-separated nested paths changed in the noisy invoice.
- `gt_products_json` / `noisy_products_json`: serialized product-line arrays.
- `evaluation_split`: present only in the noisy evaluation files; either `validation` or `test`.

## Embedding API evaluation split

The embedding scenario uses the full ground-truth corpus as the searchable index.
This is not model training: the embedding model is fixed, and the corpus is only
the database being searched.

The noisy evaluation set contains 2,100 records:

| Split | Share | Records |
|---|---:|---:|
| `validation` | 15% | 1,050 |
| `test` | 15% | 1,050 |

The split is stratified by `noise_level` and primary error type, where primary
error type is the first value in `error_types`. This keeps validation and test
close to the original distribution of noise severity and main error families.

## Error families

- OCR-like character confusion, for example `O`/`0`, `I`/`1`, `S`/`5`.
- Missing, extra, or transposed characters.
- Company-name abbreviation or legal-suffix removal.
- Buyer/seller field swaps for names and addresses.
- Date format changes, day/month swaps, and small date offsets.
- Amount rounding, small numeric deltas, and digit transposition.
- Product description shuffling inside the same invoice.
- Product description replacement with descriptions sampled from other invoices.
- Occasional missing values for high-noise records.

The default generation uses seed `42` and a target distribution of about 70% low, 20% medium, and 10% high-noise records. Structural errors such as buyer/seller swaps and product-description replacements are intentionally rarer in low-noise records and more common in medium/high-noise records.

## Embedding text and metadata files

The embedding-ready files are generated by the notebook:

```text
notebooks/prepare_embedding_jsonl.ipynb
```

They can be validated with:

```text
notebooks/check_embedding_inputs.ipynb
```

The check notebook reads `embedding_corpus.jsonl` and `embedding_queries.jsonl`
and reports record counts, schema issues, empty or very short `embedding_text`
values, text-length statistics, average product-description counts, random
examples, forbidden structured labels in the text, and `evaluation_split`
validity for query metadata.

Each output row is a JSON object with:

- `record_id`
- `source_image`
- `embedding_text`
- `metadata`

`embedding_text` is intentionally limited to semantic text useful for vector search:

- seller company name
- buyer company name
- seller address
- buyer address
- buyer country
- product descriptions

The following fields are excluded from the main embedding text and stored in `metadata` when present:

- invoice number and invoice date
- product quantities, unit prices, total prices, and line indexes
- subtotal, total, due, discounts, and other payment amounts
- VAT number and other structured identifiers
- evaluation fields such as `evaluation_split`, `noise_level`, `error_types`, and `changed_fields` for query records

This separation keeps semantic retrieval focused on text similarity, while SAP or downstream logic can use metadata for exact filters, accounting checks, fuzzy matching, and reranking.

## Generated embedding vectors

Embedding vectors are generated by:

```text
notebooks/generate_embeddings.ipynb
```

The notebook reads:

- `embedding_corpus.jsonl`
- `embedding_queries.jsonl`

and writes:

- `corpus_embeddings.jsonl`
- `query_embeddings.jsonl`

The local embedding model name, batch size, and output paths are configurable in the notebook.
They can also be configured with environment variables:

```powershell
$env:EMBEDDING_MODEL = "sentence-transformers/all-MiniLM-L6-v2"
$env:EMBEDDING_BATCH_SIZE = "64"
$env:CORPUS_EMBEDDINGS_PATH = "data/processed/corpus_embeddings.jsonl"
$env:QUERY_EMBEDDINGS_PATH = "data/processed/query_embeddings.jsonl"
```

The notebook uses `sentence-transformers`, so no API key is required. The first
run may download the model from Hugging Face; later runs use the local cache.

If needed, install the dependency with:

```powershell
pip install sentence-transformers
```

Each embedding output row keeps:

- `record_id`
- `source_image`
- `metadata`

and adds:

- `embedding_model`
- `embedding_dimension`
- `embedding`

The notebook checks required fields, empty `embedding_text` values, processed
record counts, vector dimensions, empty vectors, and preserved `record_id`
ordering.

Outputs are rewritten atomically via a temporary file and replace step. This
makes the notebook safe to rerun without appending duplicate rows or leaving a
partially written final file.

No vector database or FastAPI service is created in this step.

## Local semantic search baseline

The local semantic search baseline is generated by:

```text
notebooks/semantic_search_baseline.ipynb
```

The notebook reads:

- `corpus_embeddings.jsonl`
- `query_embeddings.jsonl`

and writes:

- `semantic_search_results.jsonl`
- `semantic_search_metrics.json`

It uses the saved vectors directly and computes cosine similarity in memory with
NumPy. No vector database, BM25, hybrid search, Weaviate, or FastAPI service is
used in this step.

The main configuration can be changed in the notebook or via environment
variables:

```powershell
$env:SEMANTIC_SEARCH_TOP_K = "5"
$env:SEMANTIC_CORPUS_EMBEDDINGS_PATH = "data/processed/corpus_embeddings.jsonl"
$env:SEMANTIC_QUERY_EMBEDDINGS_PATH = "data/processed/query_embeddings.jsonl"
$env:SEMANTIC_SEARCH_RESULTS_PATH = "data/processed/semantic_search_results.jsonl"
$env:SEMANTIC_SEARCH_METRICS_PATH = "data/processed/semantic_search_metrics.json"
```

Each row in `semantic_search_results.jsonl` preserves the query `record_id`,
`source_image`, full query `metadata`, optional `evaluation_split`, rank fields,
and a `candidates` list. Each candidate contains:

- `record_id`
- `source_image`
- `similarity_score`
- compact essential `metadata`

The correct match is defined as `query.record_id == corpus.record_id`.
`semantic_search_metrics.json` stores top-1 accuracy, top-3 accuracy, top-5
accuracy, and MRR overall and by `evaluation_split` when present.

The notebook checks processed record counts, embedding dimensions, duplicate
`record_id` values, and queries without a matching corpus `record_id`.
Outputs are written atomically through temporary files and then replaced, so the
notebook can be rerun without appending duplicate or inconsistent output.
